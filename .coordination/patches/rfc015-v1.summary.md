## Item 1 ā€” F-W1-03 [H] Remove Connected lifecycle, split into two state machines

**Owning finding**: worker-1 (confirmed by worker-2 and worker-3)
**Sections touched**:
- Ā§"Plugin Lifecycle ā€” Two Separate State Machines" (lines 92ā€“150) ā€” entirely rewritten
- Ā§"Events (Marketplace Layer)" (lines 298ā€“404) ā€” `PluginConnected` removed;
  `PluginInstanceReady`, `PluginInstanceStopped`, `PluginCredentialsVerified`
  added; `PluginEnabledForProject` extended with `signal_allowlist` and
  `signal_capture_override`; `SignalProjectedToGraph` and
  `SignalIngestedToMemory` added
- Ā§"Commands (Marketplace Layer)" (lines 405ā€“444) ā€” `ConnectPlugin` removed;
  `VerifyPluginCredentials` added as ephemeral diagnostic;
  `EnablePluginForProject` extended
- Ā§"Process Instances by Credential Scope" ā€” unchanged but now consistent with
  the top-level lifecycle rewrite
- Ā§"Decision" bullets (lines 644ā€“663) ā€” updated to remove "connected" from the
  canonical lifecycle description and add eager/lazy spawn policy
- Ā§"Integration Tests" tests 4, 4a, 4b, 10 ā€” rewritten to exercise the new
  lifecycle (lazy spawn on first use for tool-only plugins, eager spawn for
  SignalSource plugins, ephemeral verification with no lifecycle commit)

**Design decisions baked into the patch**:
1. Marketplace lifecycle: `Listed ā†’ Installing ā†’ Installed ā†’ EnabledForProject
   ā†’ DisabledForProject ā†’ Uninstalled`. No `Connected` state. No `POST /connect`
   endpoint. Rationale: the Q2 resolution (per-project credentials ā†’ per-scope
   processes) makes a single "is this plugin connected" answer meaningless when
   the same plugin can have zero, one, or many running instances simultaneously.
2. Process-instance lifecycle: `Spawning ā†’ Ready ā†’ Draining ā†’ Stopped`,
   `Failed` on error, keyed per `(plugin_id, credential_scope_key)`.
3. Spawn policy:
   - Tool-only plugins ā†’ **lazy** spawn on first tool invocation. No pre-warm
     after restart.
   - Plugins declaring at least one `SignalSource` capability ā†’ **eager** spawn
     of the tenant-default scope at first `EnablePluginForProject`. Project-
     specific credential-scoped instances still spawn lazily. This exception is
     load-bearing: worker-3's catch that without it, a webhook arriving after
     restart has no listener and is either 504'd or lost.
4. Credential verification is an **ephemeral diagnostic** via
   `POST /v1/plugins/:id/verify` (`VerifyPluginCredentials` command). It spawns
   a transient instance, runs `post_install_health_check`, and stops. It does
   NOT commit any persistent lifecycle state. This preserves worker-1 and
   worker-2's "keep an operator verification action, but not a lifecycle phase"
   refinement.
5. `PluginConnected` renamed to `PluginInstanceReady { plugin_id,
   credential_scope_key, reason: EagerSignalSource | LazyFirstInvocation |
   Restart, at }`.

## Item 2 ā€” F-W3-04 [M] Signal Knowledge Capture (cross-cut 015/017/022)

**Owning finding**: worker-3 (refined by worker-1 and worker-2)
**Sections touched**:
- Ā§"Security Model ā†’ Signal Knowledge Capture" (lines 483ā€“527) ā€” **new section**
- Ā§"Per-Project Signal Routing" (line 479) ā€” extended with `signal_allowlist`
  note
- Ā§"Relationship to Existing Systems" (lines 539ā€“556) ā€” 3 new rows for
  `cairn-graph::event_projector`, `cairn-memory::IngestService`, and
  `cairn-evals::EvalRunService` (reserved)
- Events enum ā€” `SignalProjectedToGraph` and `SignalIngestedToMemory` added

**Design decisions baked into the patch**:
1. Capture hints live on the **per-SignalType** `SignalSource` capability, not
   plugin-globally. Worker-1's refinement: a plugin can declare different
   behavior for `github.issue.opened` (ingest) vs `github.pull_request.synchronize`
   (skip). Worker-3 endorsed this placement.
2. **Split defaults**: graph projection default ON (cheap ā€” just node and edge
   inserts via existing `event_projector`), memory ingestion default OFF
   (expensive ā€” chunking + embedding + indexing). Worker-3's rationale: the
   cost profiles are different enough that a single default is wrong.
3. Capture runs **asynchronously off the durable signal event spine**
   (`SignalReceived` runtime events), not inline with webhook acceptance or
   plugin emit. Worker-2's critical refinement: keeps RFC 002 event-sourcing
   invariants intact and aligns with RFC 022's `(trigger_id, signal_delivery_id)`
   dedup ledger so replays/retries don't double-embed or double-project.
4. Per-project override via `PluginEnablement.signal_capture_override`:
   compliance-sensitive projects can force both off even when the plugin
   declared them on.
5. Memory ingestion writes via `source_id = plugin:{plugin_id}:signal:{delivery_id}`
   so ingested chunks remain deletable via the existing retention mechanism.
6. Cross-reference to RFC 020 for the ordering invariant: signal router must
   append `SignalProjectedToGraph` / `SignalIngestedToMemory` events (and the
   underlying ingest writes) **before** emitting the public `SignalReceived`
   event to downstream subscribers. Detailed recovery invariant deferred to the
   RFC 020 pass.

- Ā§"Plugin Descriptor" `PluginCategory` enum (lines 176ā€“182) ā€” `EvalScorer`

- Ā§"Non-Goals" (lines 565ā€“579) ā€” new explicit entry cutting the capability

- Ā§"Relationship to Existing Systems" ā€” `cairn-evals::EvalRunService` row

  manifests declaring it are rejected at install time"
- Ā§"Integration Tests" test 11 ā€” new test for install-time manifest rejection

**Design decisions baked into the patch**:
1. Cut from v1 scope, keep the `PluginCapability::EvalScorer` enum variant and

2. **Install-time reject** (worker-3's addition): manifests declaring

   non-goal. Silent accept would let plugin authors ship broken features.
3. Internal eval scorers shipping as cairn-rs code (using `PluginDimensionScore`
   and `PluginRubricScorer` directly) are unaffected ā€” no churn on existing
   cairn-evals types.

   decision bullet, integration test coverage.

## Item 4 ā€” F-W2-07 [M] Network egress v1 downgrade

**Owning finding**: worker-2
**Sections touched**:
- Ā§"Product Goals" goal #4 (line 54) ā€” rewritten to acknowledge "declared but
  not enforced in v1"
- Ā§"Security Model ā†’ Network Egress" (lines 533ā€“538) ā€” rewritten to make the
  v1 stance explicit ("declared and displayed only, enforcement deferred")
- Ā§"Non-Goals" ā€” new bullet making the v1 non-enforcement explicit

**Design decisions baked into the patch**:
1. Goal #4 previously promised "Configure a plugin's runtime settings (network
   egress allowlist, timeout, concurrency) without editing files", implying
   egress is enforceable. v1 security model said it was advisory only. This was
   a direct contract mismatch (worker-2's catch).
2. Downgraded v1 promise: declared hints are stored and displayed to the
   operator; `required_network_egress` list appears in the install wizard and
   per-plugin drill-in view; **enforcement is deferred to a future plugin
   sandboxing RFC**. This aligns with RFC 016's v1 stance (records but does
   not enforce network policy).
3. Operators needing stronger guarantees in v1 should run plugins under
   external process sandboxing (systemd, container network policies) outside
   cairn-app's control ā€” explicitly noted in the Security Model section.

**Owning finding**: worker-1
**Sections touched**:

**Design decisions baked into the patch**:

   the existing project-enablement filter, before reaching subscribers,

**Design decisions baked into the patch**:
1. `prompt_tools_for` only filters `Core | Registered` tiers ā€” `Deferred` is
