# Research: Obsidian Integration for Cairn

Date: 2026-04-14
Status: research complete

## Executive Summary

Obsidian can be integrated programmatically via the **Obsidian Local REST API** community plugin, which provides a full CRUD HTTP API over vault files. Authentication is API-key-based (Bearer token). There are no native webhooks -- change detection requires filesystem polling or Obsidian plugin-side event forwarding. For a Cairn integration plugin, the Local REST API is the primary interface. If real-time push notifications are a hard requirement, **Notion** is a stronger candidate (it has native webhooks as of 2025).

---

## 1. Does Obsidian Have a REST API?

**No built-in REST API.** Obsidian is a local-first desktop app with no server component.

Two programmatic interfaces exist:

### 1a. Obsidian URI Scheme (`obsidian://`)

- Desktop-only protocol handler
- Actions: `open` (file/vault), `new` (create note), `search` (open search), `hook` (various plugins)
- Format: `obsidian://action?vault=VaultName&file=path/to/note&content=...`
- Limitations: requires Obsidian running on the same machine, no response/return value, no search results, no read capability, triggers UI actions only
- **Verdict: not useful for agent integration** (no read, no programmatic response)

### 1b. Obsidian Advanced URI Plugin

- Community plugin extending the URI scheme
- Adds: write to specific heading, frontmatter editing, search-and-replace, command execution, clipboard integration
- Still URI-based (fire-and-forget, no response)
- **Verdict: useful for human automation, not for agent API calls**

### 1c. Obsidian Local REST API Plugin (the real answer)

- Community plugin by coddingtonbear
- GitHub: https://github.com/coddingtonbear/obsidian-local-rest-api
- **This is the integration target for Cairn.**

## 2. Obsidian Local REST API Plugin -- Full Details

### Installation

- Install from Obsidian Community Plugins marketplace
- Or manually: download release into `.obsidian/plugins/obsidian-local-rest-api/`
- Operator must enable the plugin in Obsidian settings

### Server Details

| Property | Value |
|----------|-------|
| Default port | 27124 |
| Protocol | HTTPS (self-signed certificate) |
| Base URL | `https://127.0.0.1:27124` |
| Certificate endpoint | `GET /obsidian-local-rest-api.crt` |
| OpenAPI spec | `GET /openapi.yaml` |
| Health check | `GET /` (no auth required) |

### Authentication

- **Mechanism**: API key via Bearer token
- **Header**: `Authorization: Bearer <api-key>`
- **Key generation**: auto-generated by plugin on first enable, visible in plugin settings
- **One key per vault** (can be regenerated in settings)
- **No OAuth, no user-level auth** -- single shared key

### Self-Signed Certificate

The plugin uses HTTPS with a self-signed cert. Clients must either:
- Trust the cert (downloadable at `/obsidian-local-rest-api.crt`)
- Disable TLS verification (e.g., `curl -k` or `reqwest::Client` with `danger_accept_invalid_certs`)
- For Cairn: accept the cert and pin it, or let operator configure TLS behavior

## 3. Complete API Endpoints

### System

| Method | Path | Description |
|--------|------|-------------|
| GET | `/` | Server status (no auth) |
| GET | `/openapi.yaml` | OpenAPI spec |
| GET | `/obsidian-local-rest-api.crt` | TLS certificate |

### Vault File Operations (`/vault/{path}`)

| Method | Path | Description |
|--------|------|-------------|
| GET | `/vault/{path}` | Read file content (markdown or JSON metadata) |
| PUT | `/vault/{path}` | Create or replace file |
| POST | `/vault/{path}` | Append content to file |
| PATCH | `/vault/{path}` | Partial update (append/prepend/replace at target) |
| DELETE | `/vault/{path}` | Delete file |

**Targeting** -- PATCH/POST/PUT support targeting specific sections:
- By URL path: `/vault/note.md/heading/Section%20Name`
- By headers: `Target-Type: heading|block|frontmatter` + `Target: <value>`
- Operations via `Operation` header: `append`, `prepend`, `replace`
- `Create-Target-If-Missing: true` header to auto-create sections

**Response formats** (via Accept header):
- `text/markdown` -- raw note content
- `application/json` -- file metadata (frontmatter, tags, path, stat)
- Document map format available for structure inspection

### Active File Operations (`/active/`)

Same methods as vault but operate on the currently open file in Obsidian:

| Method | Path | Description |
|--------|------|-------------|
| GET | `/active/` | Read active file |
| PUT | `/active/` | Replace active file content |
| POST | `/active/` | Append to active file |
| PATCH | `/active/` | Partial update active file |
| DELETE | `/active/` | Delete active file |

### Periodic Notes (`/periodic/{period}/`)

Requires Obsidian's periodic notes plugin. Periods: `daily`, `weekly`, `monthly`, `quarterly`, `yearly`.

| Method | Path | Description |
|--------|------|-------------|
| GET | `/periodic/{period}/` | Current period's note |
| PUT | `/periodic/{period}/` | Replace current period note |
| POST | `/periodic/{period}/` | Append to current period note |
| PATCH | `/periodic/{period}/` | Partial update |
| DELETE | `/periodic/{period}/` | Delete |
| GET | `/periodic/{period}/{year}/{month}/{day}/` | Specific date's note |
| PUT/POST/PATCH/DELETE | `/periodic/{period}/{year}/{month}/{day}/` | Same ops on specific date |

### Search

| Method | Path | Description |
|--------|------|-------------|
| POST | `/search/simple/` | Full-text fuzzy search (query param: `?query=terms`) |
| POST | `/search/` | Structured search via Dataview DQL or JsonLogic |

**Simple search** returns matching files with relevance scores.
**Structured search** requires the Dataview plugin for DQL queries (e.g., `TABLE file.mtime FROM "projects" WHERE status = "active"`). JsonLogic queries work against file metadata without Dataview.

### Commands

| Method | Path | Description |
|--------|------|-------------|
| GET | `/commands/` | List all available Obsidian commands |
| POST | `/commands/{commandId}/` | Execute a command (e.g., toggle plugin, export PDF) |

### File Opening

| Method | Path | Description |
|--------|------|-------------|
| POST | `/open/{path}` | Open file in Obsidian UI (`?newLeaf=true` for new pane) |

### Tags

| Method | Path | Description |
|--------|------|-------------|
| GET | `/tags/` | List all tags with usage counts |

## 4. Event/Change Detection -- No Webhooks

**The Local REST API has no webhook, SSE, or WebSocket support.** There is no way for Obsidian to push change notifications to an external service.

### Why

Obsidian is a local desktop app. The plugin API provides vault events internally (`create`, `modify`, `delete`, `rename` on the `Vault` class), but the Local REST API plugin does not expose these as outbound HTTP notifications.

### Options for Change Detection

**Option A: Filesystem watching (recommended for Cairn)**
- Obsidian vaults are plain directories of markdown files
- Use `notify` crate (Rust) to watch the vault directory for file changes
- Pros: real-time, no polling, works even when Obsidian is closed
- Cons: requires the Cairn plugin to know the vault path on the local filesystem
- This is the most reliable approach

**Option B: Polling the API**
- Periodically `GET /vault/{path}` or use search to check for changes
- Compare file `mtime` from metadata responses
- Pros: works over network
- Cons: latency, wasted requests, no way to list "recently changed files" in one call

**Option C: Custom Obsidian plugin (future)**
- A purpose-built Obsidian plugin could listen to vault events and POST to a webhook URL
- The `obsidian-webhooks` plugin by nicolevanderhoeven attempted this but the repository appears abandoned/removed
- If Cairn needs push: build a minimal Obsidian plugin that forwards `vault.on('modify')` etc. to a configurable HTTP endpoint
- This is extra work but gives true real-time push

**Recommendation for Cairn**: Use Option A (filesystem watching) as the primary change detection mechanism, with Option B (polling) as fallback for remote vaults. Option C is a potential future enhancement.

## 5. Authentication Summary

| Aspect | Detail |
|--------|--------|
| Type | Static API key |
| Transport | Bearer token in Authorization header |
| Key count | One per vault |
| Rotation | Manual regeneration in plugin settings |
| Multi-user | No (single key, no user identity) |
| Credential storage in Cairn | Store via `CredentialService` like GitHub App keys |

For Cairn's credential wizard during marketplace install:
1. Operator pastes their Obsidian Local REST API key
2. Operator provides the vault URL (default: `https://127.0.0.1:27124`)
3. Optionally: operator provides vault filesystem path (for fs-watch mode)
4. Cairn verifies connectivity via `GET /` (no auth needed) then `GET /vault/` (auth needed)

## 6. Community Plugins That Expose Vault Data via HTTP

| Plugin | Description | Status |
|--------|-------------|--------|
| **Local REST API** (coddingtonbear) | Full CRUD API, search, commands | Active, well-maintained |
| **Advanced URI** (Vinzent03) | Extended URI scheme for automation | Active, URI-only (no HTTP) |
| **obsidian-webhooks** (nicolevanderhoeven) | Outbound webhooks on vault events | Appears abandoned/removed |
| **Obsidian Web Clipper** | Inbound only (save web content to vault) | Not relevant |

The Local REST API is the only viable HTTP interface. No other maintained plugin exposes vault data over HTTP.

## 7. Comparison: Obsidian vs Notion vs Linear as Integration Targets

| Capability | Obsidian (Local REST API) | Notion API | Linear API |
|------------|--------------------------|------------|------------|
| API type | REST (community plugin) | REST (official) | GraphQL (official) |
| Authentication | Bearer token (static key) | OAuth 2.0 + API keys | OAuth 2.0 + API keys |
| Read notes/pages | Yes | Yes | Yes (issues/docs) |
| Write notes/pages | Yes | Yes | Yes |
| Search | Yes (full-text + Dataview) | Yes | Yes |
| Append to note | Yes (POST) | Yes (append blocks) | N/A |
| Webhooks (push) | **No** | **Yes** (page.content_updated, comment.created, data_source.schema_updated, page.locked) | **Yes** (create/update/delete on issues, comments, projects) |
| Real-time events | Filesystem watch only | Webhook push | Webhook push |
| Multi-user | No (single API key) | Yes (per-user OAuth) | Yes (per-user OAuth) |
| Cloud/hosted | **No** (local desktop only) | Yes (cloud-native) | Yes (cloud-native) |
| Works without app running | Partially (fs access yes, API no) | Yes (always on) | Yes (always on) |

### Recommendation

**For a third integration example (after GitHub), Notion is the strongest candidate:**

1. **Real webhooks** -- `page.content_updated`, `comment.created`, `data_source.schema_updated`, `page.locked` events pushed to your endpoint
2. **Official API** -- no community plugin dependency, stable and versioned
3. **OAuth 2.0** -- proper multi-user auth, fits Cairn's per-project credential model
4. **Cloud-native** -- always available, no local-app dependency
5. **Webhook verification** -- HMAC-SHA256 signature verification (similar pattern to GitHub's `X-Hub-Signature-256`)
6. **Rich CRUD** -- pages, databases, blocks, comments, search, users

**Obsidian is still worth building** because:
- It serves a different audience (developers who prefer local-first, markdown-based knowledge management)
- Filesystem watching is a viable alternative to webhooks
- The integration proves Cairn plugins can work with local tools, not just cloud SaaS
- Many AI/coding teams use Obsidian for documentation and knowledge bases

**Suggested integration order**: GitHub (done) -> Notion (webhooks, OAuth, cloud) -> Obsidian (local, polling/fs-watch, simple auth)

## 8. Cairn Plugin Design Sketch for Obsidian

Based on the GitHub reference plugin pattern (RFC 017):

### Capabilities

```toml
[plugin]
name = "obsidian"
version = "0.1.0"
description = "Obsidian vault integration -- read, write, search notes"

[[capabilities]]
type = "ToolProvider"
tools = [
    "obsidian.read_note",
    "obsidian.write_note",
    "obsidian.append_to_note",
    "obsidian.delete_note",
    "obsidian.search_vault",
    "obsidian.list_tags",
    "obsidian.list_commands",
    "obsidian.execute_command",
    "obsidian.read_daily_note",
    "obsidian.append_to_daily_note",
]

[[capabilities]]
type = "SignalSource"
signals = [
    "obsidian.note.created",
    "obsidian.note.modified",
    "obsidian.note.deleted",
    "obsidian.note.renamed",
]
```

### Tools

| Tool | API Call | Description |
|------|----------|-------------|
| `obsidian.read_note` | `GET /vault/{path}` | Read note content and metadata |
| `obsidian.write_note` | `PUT /vault/{path}` | Create or replace a note |
| `obsidian.append_to_note` | `POST /vault/{path}` | Append content (supports heading targeting) |
| `obsidian.delete_note` | `DELETE /vault/{path}` | Delete a note |
| `obsidian.search_vault` | `POST /search/simple/` | Full-text search across vault |
| `obsidian.list_tags` | `GET /tags/` | List all tags with counts |
| `obsidian.list_commands` | `GET /commands/` | List available Obsidian commands |
| `obsidian.execute_command` | `POST /commands/{id}/` | Execute an Obsidian command |
| `obsidian.read_daily_note` | `GET /periodic/daily/` | Read today's daily note |
| `obsidian.append_to_daily_note` | `POST /periodic/daily/` | Append to today's daily note |

### Credential Model

```
Credentials:
  - api_key: string (Bearer token from Obsidian plugin settings)
  - vault_url: string (default: https://127.0.0.1:27124)
  - vault_path: Option<string> (local filesystem path, for fs-watch mode)
  - tls_verify: bool (default: false, because self-signed cert)
```

### Signal Source Implementation

Since there are no webhooks, the plugin would use a hybrid approach:

1. **Primary**: filesystem watcher (`notify` crate) on `vault_path` if provided
   - Debounce events (Obsidian writes temp files during saves)
   - Filter to `.md` files
   - Emit `obsidian.note.{created,modified,deleted,renamed}` signals
2. **Fallback**: periodic polling via API if `vault_path` is not configured
   - Poll `GET /vault/` with metadata to check mtimes
   - Higher latency but works for remote/network vaults

### TLS Handling

The self-signed certificate is a practical concern. Options:
- Download and pin the cert at install time via `GET /obsidian-local-rest-api.crt`
- Allow operator to set `tls_verify: false` in credentials
- Store the cert in `CredentialService` alongside the API key

## 9. Key Risks and Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| Obsidian must be running for API access | Agent tools fail when Obsidian is closed | Health check before tool calls; fallback to direct filesystem read for `.md` files |
| Self-signed TLS certificate | Connection errors, certificate warnings | Pin cert at install or allow insecure mode |
| Single API key, no user identity | Cannot attribute actions to specific agents | Log tool invocations in Cairn's audit trail with agent/run context |
| No webhooks | Delayed change detection | Filesystem watching with debounce; configurable poll interval |
| Community plugin dependency | Plugin could be abandoned | The Local REST API plugin is actively maintained (regular releases); worst case, vault files are plain markdown readable from filesystem |
| Local-only access | Cannot integrate with remote/cloud Obsidian | Obsidian Sync does not expose an API; this plugin is inherently local. Document this limitation. |

## 10. Alternative: Direct Filesystem Access (No Plugin Required)

Since Obsidian vaults are plain directories of markdown files, Cairn could bypass the REST API entirely:

- **Read**: parse `.md` files directly
- **Write**: write `.md` files (Obsidian auto-detects changes)
- **Search**: full-text search via `grep`/`ripgrep` or an embedded index
- **Metadata**: parse YAML frontmatter
- **Events**: `notify` crate filesystem watcher

**Pros**: no dependency on Obsidian running, no plugin installation required
**Cons**: no access to Obsidian-specific features (commands, active file, Dataview queries, periodic notes), no guarantee of file format compatibility with future Obsidian versions

This could be a "lite mode" fallback when Obsidian is not running.
